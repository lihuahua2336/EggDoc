上一节我们完成了 Codex 的安装与配置。这一节继续往下走，先把 AI 编程里最常见、也最容易混在一起的几个基础概念理清楚。

这些概念包括：LLM、Prompt、Token、Context、Tool、MCP、Agent 和 Skill。它们看起来比较零散，但放到工程实践里，其实能串成一条完整链路：

```txt
用户需求 -> Prompt -> Agent -> MCP -> Tool -> Context -> LLM -> 输出结果
```

理解这条链路之后，再去看 Codex、Claude Code 或其他 AI 编程工具，就会清楚很多。

## LLM、Prompt 和 Token

先看前三个概念：LLM、Prompt 和 Token。

**LLM** 是 Large Language Model，也就是大语言模型。它是 AI 编程的基础能力。我们平时看到的“理解需求”“生成代码”“解释错误”，底层都要依赖模型完成。

**Prompt** 是提示词，也就是我们向模型表达需求的方式。

比如我们想创建一个网站项目，对模型说：

```txt
请帮我创建一个网站项目。
```

这句话就是一个 Prompt。

但模型并不是像人一样直接“看懂”这句话。大语言模型本质上是一个巨大的数学函数，内部运行的是大量矩阵运算。它处理的是数字，输出的也是数字。自然语言进入模型之前，会先被拆成一段段更小的单位，这就引出了第三个概念：**Token**。

Token 是模型处理语言时使用的基本单位。一个 Token 可能是一个字、一个词，也可能是几个字符组成的片段。比如“网站项目”这几个字，在不同模型里可能会被拆成不同的 Token，具体取决于模型使用的分词算法。

可以把模型处理语言的过程简化成三步：

1. 把我们输入的文字拆成 Token。
2. 模型根据已有内容预测下一个 Token。
3. 再把生成出来的 Token 还原成我们能读懂的文字。

所以，简单理解：

- LLM 是 AI 编程的基础能力。
- Prompt 是我们表达需求的方式。
- Token 是模型处理语言时使用的基本单位。

## Context

当我们和模型对话时，Prompt 不是孤立存在的。我们的提问、模型之前的回复、工具返回的结果、读到的文件内容，都会一起组成模型当前能看到的信息环境，这就是 **Context**，也就是上下文。

上下文决定了模型当前能看到什么，也决定了模型能基于哪些信息回答问题。

但上下文窗口不是无限大的。即使很多主流模型已经支持很长的上下文，一次较长的对话、工具结果和文件内容也会很快占掉一部分 Token 预算。这意味着我们不能把所有东西都塞进上下文。

围绕“如何在有限空间里，让模型拿到最有价值的信息”，就产生了 **上下文工程**。

上下文工程可以简单理解为：在有限的 Token 预算里，精选、筛选、压缩、排序和组织信息，让模型拿到最相关、最有价值的上下文。

在 AI 编程里，好的上下文通常包括：

- 当前任务目标
- 相关代码文件
- 已经运行过的命令和结果
- 错误信息
- 项目约定和限制
- 用户刚刚确认过的决策

上下文越清楚，模型越容易做出可靠判断。

## Tool

**Tool** 是工具。它可以理解成模型连接外部世界的方式。

如果只有 LLM，模型主要是在生成文字。但有了 Tool 之后，AI 编程工具就可以读文件、写文件、运行命令、搜索资料、访问浏览器、检查构建结果，甚至连接数据库。

比如你让 Codex 帮你修一个构建错误，它可能会：

1. 读取 `package.json`。
2. 运行构建命令。
3. 查看错误输出。
4. 打开相关源码文件。
5. 修改代码。
6. 再次运行构建验证。

这里的读文件、运行命令、修改文件，都是 Tool 带来的能力。Tool 把模型的回答扩展成了真实可执行的动作。

## MCP

讲到 Tool，就会遇到另一个概念：**MCP**。

MCP 的全称是 Model Context Protocol，也就是模型上下文协议。

如果说 Tool 解决的是“模型怎么连接外部能力”，那么 MCP 解决的是“AI 应用和外部工具之间，如何用一套统一方式连接”。

它有点像日常使用的 USB 接口。不同工具，比如数据库、文件系统、网页服务、业务系统，都可以按照 MCP 的标准提供能力。支持 MCP 的 AI 工具，就可以用比较统一的方式接入它们。

所以可以这样区分：

- Tool 是具体能力。
- MCP 是这些能力接入 AI 应用的一套标准方式。

在 AI 编程里，MCP 的价值是让工具接入更稳定、更可复用，也更容易组合。

## Agent

**Agent** 通常翻译成智能体。Codex、Claude Code 这类编程助手，都可以理解成 Agent。

如果说 LLM 是大脑，Tool 是手和脚，那么 Agent 更像是负责协调行动的项目经理。

一个 Agent 不只是把问题丢给模型然后等回答。它通常会围绕目标持续做几件事：

1. 理解用户目标。
2. 拆解任务。
3. 判断下一步该做什么。
4. 需要时调用工具。
5. 根据工具结果调整计划。
6. 循环执行，直到任务完成或遇到需要用户确认的边界。

所以我们可以把 Agent 理解成：

```txt
LLM + Context + Tool + 任务循环 + 约束 + 验证
```

这也是 AI 编程和普通聊天最大的区别之一：AI 编程工具不是只生成建议，而是会在项目里持续行动、检查和修正。

## Skill

**Skill** 可以理解成一种可复用的能力模块。

直接使用 Agent 可以完成很多复杂任务，但如果每次都临时组织流程，容易出现三个问题：

- 稳定性不够：同样的任务，每次执行步骤可能不一样。
- 消耗较大：每次都要把大量说明塞进上下文。
- 不好复用：流程、输入、输出和验证规则没有固定下来。

Skill 的价值，就是把一套固定流程封装起来，让 Agent 能像调用标准能力一样稳定执行。

一个标准的 Skill 通常包含三层：

1. 元数据层：说明什么时候应该使用这个 Skill。
2. 接口规范层：说明怎么调用、需要什么输入、会得到什么输出。
3. 执行体层：提供脚本、模板、参考资料、示例或具体操作流程。

普通 Prompt 往往只是“一次性的自然语言请求”。而 Skill 更像一个可管理、可复用、可测试、可编排的能力模块。

在 AI 编程里，Skill 可以把 LLM 的自然语言能力包装成工程系统能够调度的稳定能力单元。

## 一条完整链路

最后用一个例子把这些概念串起来。

假设现在有一个数据库，里面存了十万条用户行为日志。用户提出需求：

```txt
分析昨天用户活跃为什么下降。
```

这个需求首先会变成 Prompt。Prompt 说明分析目标、数据入口和期望输出格式。

接着 Agent 开始工作。它会拆解任务，规划需要看哪些数据、检查哪些指标，比如活跃、留存、渠道、异常日志。

然后 Agent 通过 MCP 标准化接入数据库工具，再用具体 Tool 连接数据库、执行 SQL、读取用户日志。

但原始日志不能全部塞进模型。因为 LLM 按 Token 处理信息，上下文窗口也有限。所以这里需要上下文工程：筛选、聚合、压缩数据，只保留关键指标、异常变化和典型样本。

如果项目里有一个“用户活跃分析”的 Skill，它还可以提供固定分析流程：先看活跃，再看留存，再看渠道，最后查异常日志。

整理后的内容形成最终 Context。最后，LLM 基于这份上下文生成分析报告。

整条链路就是：

```txt
用户需求变成 Prompt
Prompt 驱动 Agent
Agent 通过 MCP 调用 Tool
Skill 指导稳定流程
原始数据经过上下文工程变成 Context
LLM 基于 Context 生成结果
```

## 小结

这一节介绍了 AI 编程中的几个基础概念：

- LLM：生成和理解语言的基础能力。
- Prompt：向模型表达需求的方式。
- Token：模型处理语言的基本单位。
- Context：模型当前能看到的信息。
- Tool：连接外部世界并执行动作的能力。
- MCP：工具接入 AI 应用的统一协议。
- Agent：围绕目标持续规划、行动和验证的工作方式。
- Skill：可复用、可管理、可编排的能力模块。
