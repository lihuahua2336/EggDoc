这一节把 EggDoc 自身当作第一个完整项目来搭建。目标不是做一个炫技网站，而是做一个适合长期写教程、放指南、收录 AI 编程概念的中文文档站。

你会看到一个 AI 编程项目从“想法很散”到“可运行站点”的过程：先确定内容边界，再选技术栈，最后把内容、页面、搜索和部署串起来。

## 项目目标

EggDoc 的第一版要解决四件事：

- 用 Markdown 和 MDX 管理文章。
- 用清晰的页面结构承载指南、课程和笔记。
- 用静态搜索帮助用户在内容变多后快速找到答案。
- 能部署到 Cloudflare Pages，同时保留以后迁移到 VPS 静态托管的可能。

这意味着它不是一个带后台的 CMS，也不是一个用户社区。它是一个个人维护、公开阅读、内容优先的静态站点。

## 为什么选择 Astro

EggDoc 的核心是内容，而不是复杂交互。Astro 很适合这种场景：

- 默认输出静态页面，部署简单。
- 原生支持 Markdown、MDX 和 Content Collections。
- 可以在需要时引入 React 组件。
- 对文档站、教程站、内容站都比较轻。

如果一开始选 Next.js，也能完成这个项目，但很多能力会暂时用不上。EggDoc 当前不需要 SSR、Server Actions、登录态、数据库读取文章，也不需要长驻 Node 服务。

所以第一版选择：

```txt
Astro
+ TypeScript
+ Content Collections
+ Markdown / MDX
+ Tailwind CSS
+ shadcn/ui 风格组件
+ Pagefind
+ Cloudflare Pages
```

## 为什么 Markdown 优先，MDX 增强

大部分文章应该使用 `.md`。Markdown 更容易迁移，也更适合长期保存教程内容。

只有当文章需要更强的展示能力时，才使用 `.mdx`。例如：

- 可复用提示框
- 交互示例
- 多标签代码块
- 视频嵌入
- 产品对比组件

这篇文章使用 `.mdx`，是为了后续可以继续演示组件化内容。但在日常写作里，默认先写 Markdown。

## 初始化项目骨架

项目的基础命令是：

```bash
npm install
npm run dev
```

开发服务器启动后，就可以在浏览器里打开本地站点。

常用脚本：

```bash
npm run check
npm run build
npm run preview
```

其中：

- `check` 用来检查 Astro 和 TypeScript。
- `build` 会生成静态文件，并在构建后生成 Pagefind 搜索索引。
- `preview` 用来预览构建后的静态站点。

## 推荐目录结构

EggDoc 的目录结构围绕“内容”和“页面”分开：

```txt
src/
  content/
    guides/
    lessons/
    notes/
  components/
    article/
    layout/
    ui/
  layouts/
  lib/
  pages/
  styles/
docs/
  adr/
```

几个关键目录：

- `src/content` 放 Markdown 和 MDX 内容。
- `src/content.config.ts` 定义内容 schema。
- `src/pages` 定义 Astro 路由。
- `src/layouts` 放站点布局和文章布局。
- `src/lib/content.ts` 放内容查询、URL、排序、相关推荐等逻辑。
- `docs/adr` 记录重要架构决策。

这样做的好处是：文章写作、页面渲染、内容关系和工程文档各自有明确位置。

## 内容模型

EggDoc 当前使用三类内容：

```txt
guide
lesson
note
```

`guide` 用来解决具体任务，比如“用脚本把 Codex 接入 EggAi”。

`lesson` 属于连续课程，比如“AI 编程入门”里的每一节。

`note` 用来放工具介绍、概念解释和比较灵活的知识卡片。

示例 frontmatter：

```yaml
type: lesson
title: 用 Astro 和 shadcn/ui 搭建 EggDoc
```

这里有几个字段很重要：

- `draft` 控制是否发布。
- `featured` 控制是否在首页或推荐区出现。
- `path` 表示这篇 lesson 属于哪条学习路径。
- `order` 表示课程里的顺序。
- `tags` 用于标签页、相关推荐和搜索发现。

## 页面骨架

第一版页面不需要复杂，但要把主要入口搭出来：

```txt
/
/eggai/
/eggai/cursor/
/learn/
/learn/build-eggdoc/
/notes/
/notes/what-is-mcp/
/tags/astro/
/search/
```

首页负责给用户两个最清楚的入口：

- EggAi 指南
- AI 编程入门

EggAi 指南页是非线性的任务列表。用户通常带着具体问题来，比如“我想用脚本把 Codex 配到 EggAi”。

AI 编程页是线性学习路径。用户可以从第一节看到最后一节。

笔记页则收纳更灵活的工具和概念内容。

## 文章阅读体验

文章页需要比普通博客更像文档：

- 显示标题、描述、发布时间和更新时间。
- 显示标签。
- 长文章有目录。
- lesson 有上一篇和下一篇。
- 底部有相关推荐。
- 页面宽度适合长文本阅读。

这几个细节会让内容之间形成连接。否则文章多起来后，用户只能靠顶部导航来回找，很快就会迷路。

## 接入 Pagefind 搜索

内容站一旦开始变杂，搜索就会变得比分类更重要。

EggDoc 使用 Pagefind 做静态搜索：

```txt
Astro build -> dist/
Pagefind indexes dist/
Cloudflare Pages serves static files
```

构建脚本里通过 `postbuild` 生成索引：

```json
{
  "postbuild": "pagefind --site dist"
}
```

搜索页在浏览器中动态加载：

```txt
/pagefind/pagefind.js
```

这样不需要数据库，也不需要后端搜索服务。

## 使用 shadcn/ui 的边界

EggDoc 使用 shadcn/ui 的设计方式，但不要把每个区域都做成组件展示。

适合使用组件的地方：

- Button
- Badge
- Card
- Search input
- Theme toggle
- Article navigation

不适合过度组件化的地方：

- 每个段落都包卡片
- 首页做成营销页
- 为了视觉复杂度加入大量装饰

文档站最重要的是阅读节奏、信息层级和可维护性。

## Cloudflare Pages 部署

EggDoc 是静态 Astro 站点，不需要 Cloudflare adapter。

Cloudflare Pages 推荐配置：

```txt
Framework preset: Astro
Build command: npm run build
Build output directory: dist
Root directory: /
Node.js version: 22 or newer
```

仓库里可以放一个 `wrangler.toml`：

```toml
name = "eggdoc"
compatibility_date = "2026-07-03"
pages_build_output_dir = "dist"
```

如果你使用 Cloudflare 控制台部署，只需要连接 Git 仓库并填写构建配置。

如果你使用 CLI 部署，可以运行：

```bash
npm run deploy:cloudflare
```

这条命令会先构建项目，再用 Wrangler 发布 `dist`。

## 用 AI 编程时怎么推进

搭这种站点时，不要一开始就让 AI “做一个完整文档站”。更稳的方式是分阶段推进：

1. 先做规划文档，明确内容类型和技术边界。
2. 初始化项目骨架，让站点能跑起来。
3. 建内容模型，让 Markdown/MDX 能被校验。
4. 做页面和导航，让内容能被浏览。
5. 做搜索和阅读体验，让内容能被找到和串联。
6. 做部署配置，让站点能发布。
7. 再开始补真实内容。

每一步都要能运行、能检查、能提交。AI 编程的关键不是一次性生成最多代码，而是让每次推进都能被验证。

## 当前第一版还缺什么

这一版已经能作为 EggDoc 的项目起点，但还不是最终形态。

后续可以继续补：

- EggAi 真实接入教程
- Codex 安装入门的详细步骤
- MCP、Agent、上下文等概念文章
- 真实截图和视频嵌入
- 自定义 OG 分享图
- 更完整的移动端导航

但这些都可以建立在当前骨架上继续迭代。

## 小结

EggDoc 的第一版重点是打通完整链路：

```txt
内容模型 -> 页面渲染 -> 阅读体验 -> 搜索 -> 静态部署
```

当这条链路跑通后，后续最有价值的工作就不是继续堆功能，而是持续写出真正能帮助用户的内容。
